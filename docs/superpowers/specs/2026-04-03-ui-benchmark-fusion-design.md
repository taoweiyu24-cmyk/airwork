# WorkItemFlow UI 对标与融合设计

日期：2026-04-03
范围：主壳层、输入流、工作项工作台、详情页、搜索、设置、GIS 专业扩展
定位：在不丢失现有桌面能力的前提下，将 WorkItemFlow 演进为一个现代知识工作平台风格的 `工作项流转平台`

## 1. 背景

WorkItemFlow 当前 UI 已经具备多个独立产品原型：

- `Inbox` 接近传统邮件客户端 + 基础协作入口
- `WorkItems` 已进入 dashboard/card 化阶段
- `WorkItemDetail` 是功能堆叠式详情页
- `Settings` 采用桌面工具式设置入口
- `GIS` 已形成专业 GIS 子系统

问题不在于“没有界面”，而在于这些界面来自不同范式，尚未融合成一个统一产品。

本设计的目标是建立一套清晰的 UI 吸收策略：

- 明确应该向哪些优秀产品学习
- 明确哪些部分可以借鉴，哪些不能直接照搬
- 明确 WorkItemFlow 的最终融合方案
- 为后续页面级改版提供统一设计基线

## 2. 竞品角色分工

本产品不应选择单一竞品照抄，而应按角色拆分借鉴对象。

### 2.1 Notion

负责提供：

- 统一工作区感
- 强搜索、弱模块的导航方式
- 对象上下文组织能力
- 页面之间的连续感

不应照抄：

- 文档优先的产品结构
- 过于松散的块编辑器模型
- 将一切都建模为 page/database 的对象哲学

### 2.2 Linear

负责提供：

- 工作项列表密度
- 状态流转手感
- 快捷操作与响应速度
- 详情页的 action-first 设计

不应照抄：

- 纯 issue 产品模型
- 面向软件研发团队的专业术语与流程

### 2.3 Front

负责提供：

- 输入流工作台
- 团队共享收件箱思维
- 避免重复处理的协作视图
- 会话到任务的转换路径

不应照抄：

- 会话即第一公民的产品结构
- 过度以客服/支持工单场景组织全部信息

### 2.4 ClickUp / Asana

负责提供：

- 自动化和流程编排视角
- 管理者需要的概览能力
- 多视图、多筛选和批量操作

不应照抄：

- 功能过载
- 管理视图压过执行视图
- 初期就把全平台做成“大而全工作操作系统”

## 3. 总体吸收策略

WorkItemFlow 的正确融合方式应是：

- `平台骨架像 Notion`
- `任务执行体验像 Linear`
- `输入流像 Front`
- `管理能力适度借鉴 ClickUp / Asana`
- `GIS 保持专业扩展层，不进入主壳层逻辑中心`

一句话定义：

`像 Front 一样接住输入，像 Linear 一样推进工作，像 Notion 一样组织上下文。`

## 4. 能抄与不能抄

### 4.1 能抄清单

#### A. 能抄 Notion 的部分

- 顶层统一工作区感
- 对象上下文和关联信息的组织方式
- 搜索作为平台主入口
- 页面间视觉与层级一致性

#### B. 能抄 Linear 的部分

- 列表页信息密度
- 状态标签与优先级表达
- 快捷键和快速命令
- 详情页首屏布局
- 轻量但清晰的视觉层级

#### C. 能抄 Front 的部分

- 输入流的归属、处理、标记和转换
- 多人避免重复处理的界面反馈
- 邮件上下文与操作同屏
- 输入项到工作项的转化入口

#### D. 能抄 ClickUp / Asana 的部分

- 批量操作
- 管理视角筛选和视图切换
- 自动化触发概念
- 结果导向的概览卡片

### 4.2 不能抄清单

#### A. 不能抄 Notion 的“文档第一”

WorkItemFlow 的第一公民应是 `工作项`，不是文档 page。

#### B. 不能抄 Linear 的“研发 issue-only”

你的平台入口包含邮件、文件、附件、AI 摘要和 GIS，不是单纯 issue tracker。

#### C. 不能抄 Front 的“会话即平台中心”

邮件只是输入源之一，不能让整个产品结构被 inbox 绑架。

#### D. 不能抄 ClickUp 的“全部装进来”

如果把所有视图、目标、白板、自动化、文档、报表一次性塞入，会直接拖垮当前代码结构和 UI 一致性。

#### E. 不能让 GIS 变成并列第二主壳层

GIS 是高价值扩展，但不是第一阶段的产品中心。

## 5. 最终产品结构

### 5.1 顶层壳层

主壳层不应继续按技术模块组织，而应按工作流组织：

- 输入流
- 工作台
- 复盘
- 空间分析
- 设置

### 5.2 第一公民对象

平台中第一公民对象顺序应为：

1. 工作项
2. 输入项（邮件 / 文件 / 附件 / AI 提示）
3. 关联上下文（标签、联系人、位置、附件、提案、记录）
4. GIS 对象（仅在相关场景提升优先级）

### 5.3 全局入口

主入口应从“菜单”转为：

- 全局搜索 / Command Bar
- 工作流导航
- 当前对象上下文

## 6. 页面级融合方案

### 6.1 MainWindow：从模块栏变为工作流壳层

当前问题：

- 左侧图标栏是模块切换器
- 只有 Mail 有二级侧边栏
- 内容区缺少全局搜索和统一对象上下文

目标形态：

- 学 Notion：统一工作区与对象跳转
- 学 Linear：高频动作入口直接、轻量、快速

改版原则：

- 一级导航按工作流组织，不按技术模块组织
- 增加全局搜索 / 快速命令
- 内容区顶部应存在明确的 workspace 标题与当前上下文
- 邮件、工作项、GIS 仍保留原有视图，但对外以工作流名义暴露

设计结果：

- `输入流` 对应 Inbox 为主，后续可纳入 Import
- `工作台` 对应 WorkItems 为主
- `复盘` 初期可复用 WorkItems + 搜索视图
- `空间分析` 对应 GIS
- `设置` 保留

### 6.2 Inbox：从邮件列表变为输入流工作台

当前优点：

- 左侧消息列表 + 右侧阅读窗格结构稳定
- 已有同步、标记、删除、创建工作项等动作
- 已有附件区、HTML 内容区和邮件元信息

当前问题：

- 仍是邮件客户端范式，缺少“处理流”视角
- 没有归属、处理状态、已转工作项状态
- 无法直观看出哪些输入项需要处理、已处理、待确认

对标策略：

- 学 Front：让“输入流”有 ownership 和处理状态
- 学 Notion：让邮件、附件、AI 摘要、关联工作项构成统一上下文

改版方向：

- 页面标题从文件夹名升级为输入流工作台语义
- 增加 quick filters：全部输入、待处理、已转工作项、异常输入
- 每条输入项需要可见的处理状态
- 阅读窗格中强化“生成工作项 / 关联已有工作项 / AI 摘要”作为主动作

最终定位：

- Inbox 不再只是“看邮件的地方”
- 而是平台吸收外部输入并转化为工作项的主入口

### 6.3 Import：并入输入流，而不是孤立工具页

当前问题：

- `ImportView` 是一个非常工具化的页面
- 只承担“选择文件 -> 预览 -> 确认导入”
- 与 Inbox 和 WorkItems 完全割裂

设计结论：

- Import 不应长期作为一级独立页面
- 它应成为输入流工作台中的一种输入类型

改版方向：

- 第一阶段保留页面，但从导航层弱化
- 中期并入 `输入流` 视图中的 `文件输入` tab / panel
- 文件导入后直接进入“待沉淀为工作项”的流程，而不是停留在工具态

### 6.4 WorkItems：从 dashboard 变为平台主工作台

当前优点：

- 已经有统计卡片、排序、筛选、批量操作、卡片网格
- 相比传统列表已更接近现代工作台

当前问题：

- 仍是“工作项列表页”的思路，不是“主工作台”
- 缺少 `待我处理 / 最近输入 / AI 建议 / 高风险` 等聚合切片
- 搜索和筛选仍是局部控件，不是平台命令的一部分

对标策略：

- 学 Linear：列表密度、标签、操作节奏、卡片和状态
- 学 ClickUp / Asana：有限度地加入视图与管理信息

改版方向：

- 强化 quick filters 和 workspace 模式
- 将 WorkItems 作为主执行工作台，而非普通模块页
- 首屏展示“工作视图”而不是“数据统计”
- 统计卡片服务于动作，不只是展示数字

最终定位：

- 用户进入平台后，除输入流外，最常驻的页面应是 WorkItems

### 6.5 WorkItemDetail：改为 action-first 决策页

当前问题：

- AI、GIS、导出、标签、父子项、正文全部纵向堆叠
- 用户要滚动理解优先级
- 页面像“对象编辑器”，不像“处理工作项的工作台”

对标策略：

- 学 Linear：详情页首屏先给状态、摘要、下一步动作
- 学 Notion：次级上下文延后展开，避免首屏信息过载

改版方向：

- 首屏必须有：
  - 标题与状态
  - 来源与时间
  - AI 摘要 / 关键结论
  - 下一步动作建议
  - 主操作按钮
- 次级区块再放：
  - 原文
  - 标签
  - 父子项
  - 导出
  - GIS 扩展信息

最终定位：

- 这是“处理工作项”的中心，不是数据字段的堆放区

### 6.6 Search：升级为平台命令入口

当前问题：

- 搜索仍主要是页面内搜索框
- 没有成为对象跳转和平台导航入口

对标策略：

- 学 Notion：全局搜索
- 学 Linear：command menu / 快速跳转感

改版方向：

- 主壳层中放置全局搜索
- 搜索结果按对象类型分组：邮件、工作项、附件、GIS
- 搜索不只是“找数据”，还是“进入下一步工作”的入口

### 6.7 Settings：从工具设置中心转为“产品设置 + 系统设置”

当前优点：

- 已有卡片化入口
- 分类明确，易扩展

当前问题：

- 终端用户关心的产品设置和管理员关心的系统配置混在一起
- AI Provider、OAuth2、egress policy 等技术概念暴露过深

改版方向：

- 分为两层：
  - 产品设置：主题、个人偏好、默认工作方式
  - 系统设置：邮箱连接、AI Provider、目录、策略、模块
- 降低技术术语密度
- 保留卡片式入口，但增强层级表达

### 6.8 GIS：做平台扩展工作台，不做第二主产品

当前优点：

- 专业能力强
- 工具、图层、分析、导入、AI-GIS 已有较完整体系

当前问题：

- 与主平台视觉和壳层关系割裂
- 当前像一个独立 GIS 桌面软件被嵌入平台

改版方向：

- 主壳层中给 GIS 一个明确但收敛的位置：`空间分析`
- 通过统一导航、顶部标题、设计变量、对象联动体现其属于平台
- 保留专业面板与地图布局，不把主平台的轻量视觉强行套到专业 GIS 细节中

## 7. 视觉系统方向

### 7.1 基本方向

视觉目标不是“炫”，而是：

- 高信息密度
- 清晰层级
- 强上下文感
- 弱噪音
- 快速操作反馈

### 7.2 风格融合

主平台建议：

- 底色偏中性、低噪声
- 结构线弱化
- 用层级、间距、字体权重建立秩序
- 少量强调色表达状态与动作

工作项与输入流：

- 更接近 Linear 的紧凑效率感

壳层和信息组织：

- 更接近 Notion 的平衡与统一

GIS：

- 保持专业工作台风格，但共享色板、字重、状态语义

### 7.3 禁止项

- 不要把不同页面做成不同产品皮肤
- 不要为了“现代感”过度加渐变、玻璃、装饰噪音
- 不要让统计卡片抢走主操作区的注意力
- 不要让每个页面都有一套新的按钮、标签、间距逻辑

## 8. 设计原则

后续所有 UI 改版统一遵守以下原则：

1. 工作流优先于模块
2. 工作项优先于文档
3. 输入流优先于工具入口
4. 搜索优先于层级跳转
5. 首屏优先展示摘要和动作，而不是全部字段
6. GIS 保持专业，但必须纳入平台统一壳层
7. 所有新增 UI 都必须解释其服务的是主流程哪一段

## 9. 实施优先级

### 第一优先级

- MainWindow 工作流化
- Inbox 向输入流工作台演进
- WorkItems 成为平台主工作台
- 全局搜索前置

### 第二优先级

- WorkItemDetail action-first 改版
- Settings 分层
- Import 并入输入流

### 第三优先级

- GIS 平台化整合
- 管理视图与自动化能力扩展

## 10. 最终结论

WorkItemFlow 不应该成为：

- 一个缩小版 Notion
- 一个桌面版 Linear
- 一个带邮件功能的 Front
- 一个简化版 ClickUp

它应该成为一个有自己中心的产品：

`以工作项为中心，以输入流为起点，以 AI 和搜索增强处理效率，以 GIS 提供专业扩展能力。`

在 UI 上，这意味着：

- 用 Notion 解决“平台怎么组织”
- 用 Linear 解决“工作项怎么推进”
- 用 Front 解决“输入流怎么吸收”
- 用 ClickUp / Asana 只补管理与自动化，不主导主界面
