# WorkItemFlow 平台级复盘与路线图设计

日期：2026-04-03
范围：项目级架构复盘、功能实现缺口识别、前端 UI 审计、中长期产品路线图
定位：`工作项流转平台`，GIS 为专业能力模块，而不是产品主轴

## 1. 背景与目标

当前仓库已经具备较完整的模块骨架：

- `Wif.App`：Avalonia 桌面 UI
- `Wif.Core`：编排与应用服务
- `Wif.Domain`：实体、仓储接口、枚举和值对象
- `Wif.Data`：SQLite / EF Core / FTS5 / 仓储实现
- `Wif.AI`：OpenAI-compatible 接入、Prompt、Egress Policy
- `Wif.Mail`：IMAP/SMTP、OAuth2、同步
- `Wif.Docs`：文档抽取与导出
- `Wif.Search`：FTS + DuckDB 混合检索
- `Wif.Gis`：地图、图层、空间分析、AI-GIS 工具

但产品层面存在三套并行“真相源”：

1. `CLAUDE.md` 与解决方案结构所声明的目标能力
2. 当前代码真正已经连通的功能
3. `docs/superpowers` 与 GIS/UI 文档中规划过但尚未完全落地的能力

本设计的目标不是再写一份泛化愿景，而是建立一套平台级审计框架，用于回答四个问题：

- 当前哪些能力真正可交付
- 哪些能力只是部分打通或 UI/文档占位
- 哪些结构问题会阻碍中长期演进
- 应该如何按产品化优先级推进，而不是按模块平推

## 2. 产品定位与主流程

### 2.1 产品定位

WorkItemFlow 应被定义为一个面向知识工作者和业务处理人员的 `工作项流转平台`：

- 邮件、文件、手工输入是入口
- AI 分析负责摘要、行动项提取、分类与建议
- 工作项是核心持久化对象
- 检索、状态推进、结构化复盘是主闭环
- GIS 是专业场景扩展，用于位置、空间数据、地图驱动分析与执行

### 2.2 中枢业务链路

本轮路线图评审统一围绕以下主流程：

`邮件/文件进入 -> AI 分析 -> 生成工作项 -> 分派/跟进 -> 检索/复盘`

所有模块、UI 与质量治理都应回答它们如何服务这条链路。不能直接增强主流程的能力，不应进入平台第一优先级。

## 3. 审计方法

### 3.1 三源对照

每个能力点均以三类证据交叉验证：

- 架构声明：`CLAUDE.md`、`Wif.sln`、模块边界
- 代码事实：View、ViewModel、Service、Repository、测试
- 规划承诺：`docs/superpowers/plans/*`、`docs/gis-ui-interface-map.md`

### 3.2 审计维度

平台审计分为五个面：

1. 主流程能力审计
2. 模块边界审计
3. 前端交互审计
4. 视觉系统审计
5. 质量治理审计

### 3.3 状态分级

不使用简单的“已做/未做”，统一使用以下五档：

- `已完成`：入口、后端行为、状态反馈、数据持久化、回显链路全部闭环
- `半完成`：真实能力存在，但缺关键 UI、异常处理、联动或可验证性
- `占位`：只有菜单/命令/状态提示/说明性逻辑，没有产品级闭环
- `缺失`：架构或规划中应存在，但代码没有对应实现
- `失真`：文档声称已实现，但代码实际并未达到“已完成”标准

### 3.4 “已实现”判定标准

某功能只有同时满足以下条件，才可标记为 `已完成`：

- 有稳定入口
- 有真实后端行为
- 有明确状态反馈
- 结果可以在同一流程中被再次验证或消费

这一定义用于清理“按钮存在即视为完成”的误判。

## 4. 当前平台能力评估

### 4.1 主流程能力矩阵

#### A. 入口层：邮件 / 文件 / 手工输入

评估结果：`半完成`

已确认能力：

- 邮件账户接入、同步、读取、附件后台处理已具备基础能力
- 从邮件创建工作项已具备路径
- 手工创建工作项已具备路径
- 文档处理与导出模块已经具备基础骨架

主要证据：

- `src/Wif.App/ViewModels/InboxViewModel.cs`
- `src/Wif.App/ViewModels/WorkItemsViewModel.cs`
- `src/Wif.App/ViewModels/SettingsViewModel.cs`
- `src/Wif.Docs/*`

主要缺口：

- “文件进入 -> 自动沉淀为工作项”的产品路径不清晰，导入页面与工作项沉淀没有形成统一入口
- 入口层仍按“邮箱 / 导入 / 工作项”模块分割，而不是按“待处理输入流”统一组织
- 账户配置、授权、同步、附件抽取均存在，但缺少统一的产品状态中心和故障恢复体验

#### B. AI 分析层

评估结果：`半完成`

已确认能力：

- AI Provider / Profile / Prompt / Egress Policy 已具备骨架
- 工作项详情页可触发分析并载入 proposals
- Settings 已支持 AI Profile 管理

主要证据：

- `src/Wif.AI/*`
- `src/Wif.Core/Services/AnalysisService.cs`
- `src/Wif.App/ViewModels/WorkItemDetailViewModel.cs`
- `src/Wif.App/ViewModels/SettingsViewModel.cs`

主要缺口：

- AI 分析仍停留在详情页局部能力，没有成为平台级批量处理和入口分拣能力
- 缺少 AI 结果的“可信度、来源、审批状态、回滚语义”统一模型
- AI 配置与业务策略解耦不足，平台未形成“默认分析策略 / 场景策略 / 模块策略”

#### C. 生成工作项层

评估结果：`半完成`

已确认能力：

- 邮件可生成工作项
- 手工可创建工作项
- 工作项可修改状态、设置父子关系、标签、导出

主要证据：

- `src/Wif.App/ViewModels/InboxViewModel.cs`
- `src/Wif.App/ViewModels/WorkItemsViewModel.cs`
- `src/Wif.App/ViewModels/WorkItemDetailViewModel.cs`

主要缺口：

- 缺少统一的“创建工作项策略”，例如来自邮件、附件、AI 摘要、导入文件的创建规则不统一
- 缺少“去重 / 合并 / 同源聚合 / 批量生成”的平台能力
- 工作项模型偏单体记录，缺少负责人、协作关系、SLA、时间维度、处理记录等产品化字段与流程约束

#### D. 分派 / 跟进层

评估结果：`占位到半完成之间`

已确认能力：

- 状态流转、父子关系、标签存在
- 批量状态修改存在

主要证据：

- `src/Wif.App/ViewModels/WorkItemsViewModel.cs`
- `src/Wif.App/ViewModels/WorkItemDetailViewModel.cs`

主要缺口：

- 缺少负责人 / 协作人 / 队列 / 看板 / SLA / 催办 / 处理记录
- 缺少评论、时间线、审计日志、通知与订阅
- 跟进行为还没有从“对象编辑”升级为“流程推进”
- 详情页没有形成完整作业台，只是信息堆叠页

#### E. 检索 / 复盘层

评估结果：`半完成`

已确认能力：

- Repository 搜索存在
- Hybrid SearchService 已具备能力入口
- FTS5 + DuckDB 架构已经声明且搜索模块与测试存在

主要证据：

- `src/Wif.Core/Services/SearchService.cs`
- `src/Wif.Search/*`
- `tests/Wif.Search.Tests/*`

主要缺口：

- 前端主要仍调用普通搜索，没有把混合检索明确暴露为平台级体验
- 检索结果没有统一的 facet、排序、保存搜索、复盘视图
- “复盘”没有落为一个独立产品能力，当前更像搜索而不是总结、趋势与闭环分析

### 4.2 GIS 在平台中的定位评估

评估结果：`专业扩展层，当前实现度高于产品整合度`

已确认能力：

- GIS 子系统具备丰富命令、导入、图层、空间分析、栅格操作、AI-GIS 连接能力
- UI、地图交互、图层与工具体系已经形成较大子系统

主要证据：

- `src/Wif.App/ViewModels/GisViewModel.cs`
- `src/Wif.App/Views/GisView.axaml`
- `src/Wif.Gis/*`
- `tests/Wif.Gis.Tests/*`

主要问题：

- GIS 的工程体量和 UI 密度已经接近独立产品，而主平台工作流还未成熟
- GIS 文档与代码状态存在失真，部分功能属于说明性实现、文本结果模拟或占位命令
- GIS 与工作项主流程的整合深度不足，目前更多是“从详情页进入 GIS”，而不是“GIS 服务于工作项处理闭环”

结论：

- GIS 不应继续作为平台主轴扩张
- 应重构为可插拔的专业工作台
- 平台主流程完成前，GIS 新功能应以“服务工作项闭环”为准入条件

## 5. 架构问题评估

### 5.1 当前优点

- 模块划分总体清晰
- Core / Domain / Data / AI / Mail / Docs / Search / Gis 具备明确命名空间边界
- 解决方案结构与业务能力基本一致
- 搜索、GIS、文档等模块已经独立成项目，避免所有能力挤在 App 层

### 5.2 关键结构风险

#### A. ViewModel 过重

风险等级：高

代表案例：

- `src/Wif.App/ViewModels/GisViewModel.cs`
- `src/Wif.App/ViewModels/WorkItemDetailViewModel.cs`
- `src/Wif.App/ViewModels/SettingsViewModel.cs`

问题：

- 命令、状态、文件选择、业务编排、错误处理、结果文本拼装混在同一类中
- UI 层承担了过多产品编排责任
- 导致测试成本升高、复用困难、局部改动风险扩大

建议：

- 引入场景编排器或 feature-level application service
- 将“命令集”“视图状态”“领域编排”“外部交互”拆开

#### B. 文档可信度不足

风险等级：高

代表案例：

- `docs/gis-ui-interface-map.md`
- `src/Wif.App/ViewModels/GisViewModel.cs`

问题：

- 文档标记“已实现”的能力，代码中仍存在 `功能开发中`、`待集成`、说明性文本结果
- 规划文档和实现状态未建立同步机制

建议：

- 引入“文档状态必须由代码证据支持”的规则
- 所有 spec / plan / capability map 应区分：设计完成、代码完成、UI 完成、验证完成

#### C. 平台入口割裂

风险等级：高

问题：

- 主窗口以模块切换为中心，而非工作流中心
- 邮箱、导入、工作项、GIS、设置是并列模块
- 用户必须自行跨模块拼接流程

主要证据：

- `src/Wif.App/MainWindow.axaml`
- `src/Wif.App/MainWindow.axaml.cs`

建议：

- 主导航应从“模块”转向“工作流”
- 例如：输入流、待处理、处理中、复盘、空间分析、系统设置

#### D. 搜索能力未成为工作台能力

风险等级：中高

问题：

- Search 架构存在，但前台仍主要是局部搜索框
- 检索没有成为统一工作入口

建议：

- 将搜索提升为全局命令中心与对象导航入口
- 让搜索结果可联动到邮件、工作项、附件、GIS 结果

## 6. 前端 UI / 交互审计

### 6.1 当前整体判断

当前 UI 已经从“纯表单桌面应用”向“信息型工作台”迈出一步，但总体仍处于：

- 信息结构分散
- 组件视觉语言不统一
- 模块感强于工作流感
- 页面间层级不一致
- 高价值操作缺少清晰主次关系

### 6.2 主要界面问题

#### A. 主窗口信息架构仍是“模块盒子”

问题：

- 左侧主导航仅按模块图标切换
- 二级侧边栏只有 Mail 有内容，其余模块缺少上下文导航
- 内容区没有统一的“对象上下文”与“全局命令”区域

影响：

- 难以建立现代知识工作平台的连续感
- 用户切换模块时上下文被打断

#### B. WorkItems 列表页已有 modern dashboard 雏形，但还不是平台级收件/处理中心

主要证据：

- `src/Wif.App/Views/WorkItemsView.axaml`
- `src/Wif.App/ViewModels/WorkItemsViewModel.cs`

问题：

- 卡片化改造已经完成，但更多是视觉重排，而不是任务工作台重构
- 缺少“待我处理 / 今日 / AI 建议 / 最近输入 / 高风险”这类知识工作平台常见聚合视图
- 搜索、筛选、批量操作还停留在单页面控制条，没有进入统一命令体系

#### C. WorkItem 详情页是纵向堆叠页，不是决策与执行工作台

主要证据：

- `src/Wif.App/Views/WorkItemDetailView.axaml`
- `src/Wif.App/ViewModels/WorkItemDetailViewModel.cs`

问题：

- AI 分析、GIS、标签、父子项、导出、状态变更全部直接堆叠
- 缺少区块优先级、操作主次、时间线与上下文摘要
- 缺少“下一步动作”设计

#### D. Settings 页承担过多技术配置暴露

问题：

- 对终端用户而言，AI、邮件、目录、OAuth2、egress policy 的层级混合过深
- 缺少“产品设置”与“系统配置”的区分

#### E. GIS UI 已形成专业桌面 GIS 风格，但与主平台视觉割裂

问题：

- GIS 页面是经典 GIS 软件布局
- 与 WorkItems / Mail / Settings 的视觉语言和交互范式明显不同

结论：

- GIS 保持专业性是合理的
- 但需要通过导航、命令、色彩、组件和入口方式，体现其是平台扩展，而非第二个独立产品

### 6.3 对标“现代知识工作平台”的 UI 原则

后续 UI 设计统一遵循以下原则：

1. 工作流优先于模块
2. 搜索优先于菜单
3. 对象上下文优先于孤立页面
4. 摘要、状态、下一步动作必须在首屏
5. 二级信息延迟展开，不与主决策区竞争
6. 同类交互在不同页面保持同一视觉与行为模型

## 7. 中长期目标架构

### 7.1 目标产品结构

平台应收敛为三层：

#### 层 1：平台主流程层

包含：

- 输入流中心
- 工作项工作台
- 搜索与复盘
- 通知与状态中心

#### 层 2：能力服务层

包含：

- AI 分析
- 文档抽取
- 邮件接入
- 搜索索引
- 身份 / 联系人 / 规则

#### 层 3：专业扩展层

包含：

- GIS 工作台
- 空间分析
- AI-GIS 自动化执行

### 7.2 目标前端结构

App 前端应从“每个模块一个大 ViewModel”迁移为：

- Shell 层：全局导航、全局搜索、通知、命令入口
- Workspace 层：输入流、工作项、复盘、GIS 等工作台
- Feature 层：列表、详情、批量操作、审阅、执行器
- Dialog / Panel 层：辅助操作与局部配置

## 8. 路线图设计

### Phase 1：平台真相统一与主流程闭环

目标：

- 统一能力状态
- 清理占位与失真
- 让主流程形成可验证闭环

重点任务：

- 建立平台能力矩阵
- 标记已完成 / 半完成 / 占位 / 缺失 / 失真
- 清理 GIS 文档与实现状态不一致
- 明确入口层如何沉淀为工作项
- 将搜索升级为主流程能力而不是附属框

验收标准：

- 主流程每一段都有代码证据与 UI 入口
- 任何”已实现”能力都可被验证

> **执行时真相源**: Phase 1 能力矩阵已发布至 [`docs/superpowers/platform-capability-matrix.md`](../platform-capability-matrix.md)，所有能力状态以该文档为准。GIS 接口映射文档 [`docs/gis-ui-interface-map.md`](../../gis-ui-interface-map.md) 已同步修正失真条目。

### Phase 2：产品信息架构与统一工作台

目标：

- 让平台从“模块集合”升级为“统一工作流工作台”

重点任务：

- 重做主导航与二级导航
- 建立全局搜索 / 全局命令入口
- 将 WorkItems 升级为平台主工作台
- 重构 WorkItemDetail 为摘要 + 动作 + 上下文布局

验收标准：

- 用户可在不理解内部模块结构的前提下完成主流程
- 页面间上下文切换成本显著下降

### Phase 3：协作与治理能力补齐

目标：

- 把“对象编辑”升级为“流程推进”

重点任务：

- 引入负责人、队列、关注、处理记录、时间线
- 增加规则、通知、审计与异常恢复
- 建立 AI 审批、回滚与来源说明机制

验收标准：

- 工作项支持团队协作和过程追踪
- AI 行为具备审批、执行、复核边界

### Phase 4：GIS 专业扩展层整合

目标：

- GIS 以插件化工作台方式服务平台，而不是反客为主

重点任务：

- 重构 GIS 入口与工作项联动模式
- 明确哪些 GIS 功能服务主流程，哪些保持专业子系统
- 对 GIS 文档、能力状态和测试建立单独治理基线

验收标准：

- GIS 与主平台共享导航、对象上下文和设计系统
- GIS 专业能力与平台能力边界清晰

## 9. 质量治理方案

### 9.1 代码质量

- 对超大 ViewModel 建立拆分标准
- 新增功能优先沉到 service / coordinator，而不是直接堆进 ViewModel
- 为主流程建立 feature-level tests，而不只保留底层单元测试

### 9.2 文档质量

- 设计文档必须区分“设计状态”和“实现状态”
- UI 地图、能力矩阵和 plan 文档必须引用代码证据
- 禁止以菜单存在、命令存在代替功能完成

### 9.3 交付质量

- 每个阶段都必须有明确的验收矩阵
- 评审输出应包含：能力状态、风险、验证方式、UI 影响
- 所有面向用户的高价值功能必须同时通过“代码完成 + UI 可达 + 可验证”三项检查

## 10. 本轮复盘输出物定义

在本设计之后，后续计划应至少生成以下产物：

1. 平台能力矩阵
2. 主流程分段评审表
3. UI 缺口清单与重构优先级
4. 架构风险清单
5. 分阶段执行计划

## 11. 已确认的高优先级结论

1. WorkItemFlow 的中长期方向应坚定定位为 `工作项流转平台`
